## 🧠 1. TikTok이 주목하는 디바이스 신호

TikTok은 **행동**과 **시스템**의 종합적인 신호를 평가합니다:

- 디바이스 지문(SoC, 메인보드, 빌드 태그, 센서)
- 미디어 파이프라인(하드웨어 코덱, 프레임 타임스탬프)
- 네트워크 스택 및 IP 평판
- 입력 역학(클릭 궤적, 스와이프 곡선, 타이핑 리듬)

> 에뮬레이터는 종종 **합성/누락된 신호**를 노출하여 신뢰도를 낮추거나 추가 검토를 트리거합니다.

---

## 📱 2. 실제 디바이스 = 더 강한 신뢰성

| 신호 계층 | 에뮬레이터/가상 환경 | 실제 Android |
|---|---|---|
| Build/ro.* 속성 | 일반적이고 반복적 | **OEM과 일치하고 다양함** |
| 센서 | 희소/시뮬레이션 | **자이로, 가속도, 자기, 광센서** 및 자연 노이즈 포함 |
| 미디어/코덱 | 소프트 코덱 문제 발생 쉬움 | **하드 코덱** 타임스탬프 안정적 |
| 전원/온도 제어 | 곡선이 "너무 평탄" | **실제 스로틀링 및 대기 주기** |
| 입력 타이밍 | 기계적 간격 | **사람처럼 무작위** |

**결과:** 실제 디바이스는 **신뢰할 수 있는 자연스러운 차이**를 생성하여 실제 사용자에 더 가깝습니다.

---

## 🎬 3. 미디어 파이프라인과 추천 피드(FYP)

- 하드웨어 코덱은 **프레임 드롭/오디오-비디오 드리프트** 감소  
- 정확한 프레임 레이트 → 더 나은 **완료율/시청 시간** 진정성  
- 안정적인 타임스탬프는 **품질 점수** 및 배포 향상

> 같은 비디오라도 파이프라인이 "이상"하면 순위가 낮아질 수 있습니다.

---

## 🔐 4. 무결성 및 환경 검증

구체적인 규칙은 공개되지 않았지만 일반적인 모바일 신호에는 다음이 포함됩니다:

- 빌드 태그(예: test-keys), QEMU/VM 특성  
- 전화 스택 누락/중복 디바이스 식별자  
- 센서 부재 또는 이상, 고도로 균질한 MAC 세그먼트, adb 상태  
- 시스템 보안 상태(root/디버그 스위치)

실제 디바이스는 많은 "위장이 필요한" 위험 신호를 **자연스럽게 회피**합니다.

## ⚖️ 5. 규모 확장 시 안정성

| 지표(대표 실험) | 에뮬레이터 클러스터 | 실제 디바이스 |
|---|---|---|
| 2시간 세션 생존율 | 78–88% | **96–99%** |
| 제스처 지터 p95 | 80–120 ms | **30–60 ms** |
| 100개 게시물당 재시도 업로드 | 12–18 | **2–5** |
| FYP 푸시(동일 콘텐츠) | 낮고 변동적 | **더 높고 안정적** |

*예시일 뿐; 실제는 프록시 품질, 콘텐츠, 디바이스 상태와 관련됩니다.*

---

## 🧰 6. 실제 디바이스 모범 사례

- **실제 Android 디바이스** 고수(에뮬레이터 사용 안 함)  
- "오염된" 중고 디바이스 피하기(이전에 자동화에 사용됨)  
- 디바이스당 하나의 **주거용 프록시**(공유 VPN 사용 안 함)  
- **OEM 펌웨어** 및 패치 유지; 개발자 옵션 끄기  
- root 안 함; 지역/언어를 IP와 일치시키기

---

## 🔄 7. 에뮬레이터에서 실제 디바이스로 마이그레이션

1. 먼저 **소규모 파일럿**(10–20대)으로 KPI 검증  
2. 계정과 디바이스/프록시 **일대일 매핑**  
3. 시간차 스케줄링, **사람처럼 무작위** 도입  
4. 연결 끊김, 재시도, FYP 노출 모니터링  
5. 전원 Hub 및 두 번째 워크스테이션을 통해 **수평 확장**

## ✅ 8. 위험 관리 체크리스트

| 카테고리 | 권장사항 |
|---|---|
| 하드웨어 | 실제 Android, 건강한 케이블, 전원 Hub |
| 네트워크 | 디바이스별 주거용 IP, 공유 VPN 피하기 |
| 시스템 | 원본 펌웨어, root 안 함, 안정적인 시간대/언어 |
| 행동 | 워밍업, 사람처럼 입력, 작업 시간차 |
| 콘텐츠 | 안정적인 미디어 파이프라인; 완료율에 집중 |
| 관찰 | 세션 상태, 재시도율, FYP 커버리지 추적 |
